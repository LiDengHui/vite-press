前端动画里有一个老大难问题，几乎每个做过交互的人都踩过：

height: 0 能动画，height: auto 却常常不能顺滑过渡。

这会直接影响大量常见组件：

  • 手风琴
  • 折叠面板
  • FAQ 展开收起
  • 评论区展开
  • 筛选面板开合
  • 导航子菜单显隐

过去我们为了让这些组件“看起来能平滑展开”，常常不得不写一堆并不优雅的方案：

  • 用 max-height 硬凑
  • 用 JavaScript 测量 scrollHeight
  • 展开前后手动写入像素值
  • 利用 FLIP 技术模拟尺寸动画

这些方案能做出效果，但维护成本并不低。现代 CSS 的 interpolate-size，正是为了解决这类尺寸无法自然插值的问题。它让浏览器能够更合理地处理 auto、max-content、fit-content 这类内在尺寸值的过渡。

本文会系统讲清楚它的意义、使用思路、适用场景、与传统方案的区别，以及一组可以直接上项目的“奇淫技巧”。

一、为什么 auto 尺寸动画一直很难 ​

先看一个最常见的折叠面板：

css
.panel {
  height: 0;
  overflow: hidden;
  transition: height 0.3s ease;
}

.panel.is-open {
  height: auto;
}

很多人第一次写都会期待它顺滑展开，但现实通常是：

  • 直接跳变
  • 动画不生效
  • 展开收起不对称

原因很简单：

  • 0 是明确数值
  • auto 不是具体数值，而是布局计算结果

浏览器要做过渡，前提是能在起点和终点之间插值。但 auto 不是一个固定数字，所以传统上难以直接参与插值。

二、传统解决方案为什么不够优雅 ​

1. max-height 大法 ​

css
.panel {
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.3s ease;
}

.panel.is-open {
  max-height: 500px;
}

这招的优点是简单，缺点也很明显：

  • 你得猜一个上限
  • 内容短时动画节奏不自然
  • 内容超出上限会被截断
  • 组件复用时非常脆弱

2. JavaScript 测量 scrollHeight ​

js
panel.style.height = panel.scrollHeight + 'px';

再配合状态切换实现展开收起。

问题在于：

  • 逻辑复杂
  • 要处理初始值与回流时机
  • 字体加载或内容异步更新时容易出边界问题
  • 样式层被脚本干预较多

3. 变通动画 ​

有人会改用：

  • scaleY
  • clip-path
  • opacity + transform

这些可以做“像展开”的视觉效果，但本质上并不是真正的布局尺寸过渡。很多情况下：

  • 文本会被压缩变形
  • 布局流不跟着变化
  • 可访问性与真实交互状态不完全一致

三、interpolate-size 是什么 ​

interpolate-size 可以理解为：

告诉浏览器：允许在尺寸过渡中，对内在尺寸值进行插值处理。

最常见的使用方式是：

css
:root {
  interpolate-size: allow-keywords;
}

这表示：

  • 当涉及尺寸动画时
  • 浏览器可以尝试让 auto、min-content、max-content、fit-content 等关键字尺寸参与过渡

这类能力最大的意义就在于：

过去需要很多 JS 才能做得比较自然的尺寸动画，现在 CSS 可以原生接管更多。

四、最基础的展开收起示例 ​

HTML ​

html
<button class="faq-toggle">什么是尺寸插值？</button>
<div class="faq-panel">
  <p>尺寸插值是指让元素从明确尺寸过渡到内在尺寸时，动画能够连续而不是跳变。</p>
</div>

CSS ​

css
:root {
  interpolate-size: allow-keywords;
}

.faq-panel {
  height: 0;
  overflow: clip;
  transition: height 0.3s ease;
}

.faq-panel.is-open {
  height: auto;
}

这段代码的意义 ​

重点不在 transition 本身，而在：

  • 全局声明允许关键字尺寸参与插值
  • height: auto 不再只是“跳变目标”
  • 浏览器有机会把它作为可过渡终点处理

这就是它最核心的价值。

五、它适合解决哪些问题 ​

1. 手风琴组件 ​

最典型的应用场景，没有之一。

css
.accordion-panel {
  height: 0;
  overflow: clip;
  transition: height 0.25s ease;
}

.accordion-item[open] .accordion-panel {
  height: auto;
}

2. 导航子菜单展开 ​

css
.submenu {
  height: 0;
  overflow: hidden;
  transition: height 0.2s ease;
}

.nav-item.is-open .submenu {
  height: auto;
}

3. 评论区“展开更多” ​

css
.comment-list {
  height: 0;
  overflow: clip;
  transition: height 0.3s ease;
}

.comment-list.is-expanded {
  height: auto;
}

4. 搜索筛选面板 ​

css
.filter-panel {
  height: 0;
  overflow: clip;
  transition: height 0.24s ease;
}

.filter-panel.is-open {
  height: auto;
}

这些场景的共同点是：

  • 展开后的内容高度不固定
  • 内容可能动态变化
  • 用死板像素值很难长期维护

六、与 calc-size() 的关系 ​

现代尺寸插值讨论里，常常还会一起提到 calc-size()。你可以把它理解为：

围绕内在尺寸值进行计算与组合的补充工具。

例如在某些需要“基于内容尺寸再额外补一点空间”的场景里，它会比纯像素写法更灵活。

虽然在大多数业务场景中，interpolate-size 已经足够先解决“能不能平滑过渡”的核心问题，但了解 calc-size() 的方向很重要：

  • 未来尺寸动画会更声明式
  • 布局结果不再总要由 JS 读出来再写回去
  • CSS 能直接表达更多“内容驱动的尺寸关系”

对于日常项目，先掌握 interpolate-size 更实用；对于现代 CSS 体系理解，知道 calc-size() 是同一方向的补充即可。

七、和 transform 模拟展开有什么区别 ​

很多项目会这样写：

css
.panel {
  transform-origin: top;
  transform: scaleY(0);
  transition: transform 0.3s ease;
}

.panel.is-open {
  transform: scaleY(1);
}

这能做出“视觉上的展开”，但本质和真实尺寸动画不是一回事。

transform 方案的问题 ​

  • 元素内容会被压缩拉伸
  • 周围布局不一定同步变化
  • 高度不是实际展开
  • 可点击区域与真实视觉关系可能出现偏差

interpolate-size 的优势 ​

  • 动画的是真实尺寸变化
  • 布局流会自然跟着更新
  • 对折叠区、FAQ、评论区这类真实文档流内容更合理

所以如果目标是“真的展开内容区域”，尺寸插值通常比 transform 更符合语义。

八、渐进增强怎么做 ​

interpolate-size 属于现代 CSS 能力，项目里一定要考虑增强与回退策略。

基础版本 ​

css
.panel {
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.3s ease;
}

.panel.is-open {
  max-height: 40rem;
}

这是传统兜底。

增强版本 ​

css
@supports (interpolate-size: allow-keywords) {
  :root {
    interpolate-size: allow-keywords;
  }

  .panel {
    max-height: none;
    height: 0;
    overflow: clip;
    transition: height 0.3s ease;
  }

  .panel.is-open {
    height: auto;
  }
}

这种写法的优点是：

  • 老浏览器还能靠 max-height 工作
  • 新浏览器获得更自然的尺寸动画
  • 不需要为了兼容彻底放弃现代能力

九、工程里最推荐的使用姿势 ​

1. 优先用于内容高度不固定的折叠组件 ​

如：

  • FAQ
  • 折叠菜单
  • 详情展开
  • 配置面板
  • 评论区

2. 配合 overflow: clip 或 hidden ​

css
.panel {
  overflow: clip;
}

尺寸在过渡时，如果不裁切内容，很容易出现视觉穿帮。

3. 让状态切换尽量语义化 ​

例如：

  • .is-open
  • [open]
  • [aria-expanded="true"]

这样样式层和交互层更容易配合。

4. 动画时间不要太长 ​

尺寸变化类动画最好控制在：

  • 0.2s
  • 0.24s
  • 0.3s

过长会显得拖沓，特别是 FAQ 和过滤器这种高频操作区域。

5. 动态内容更新后要测试 ​

比如：

  • 图片异步加载
  • 字体切换
  • 文本国际化变长
  • 服务端返回内容增多

尺寸插值虽然更先进，但真实内容变化仍然需要你做完整测试。

十一、奇淫技巧 ​

技巧一：全局只开一次关键字尺寸插值 ​

css
:root {
  interpolate-size: allow-keywords;
}

如果项目里多个折叠组件都要用，这样统一声明最省事，不必每个组件重复写。

技巧二：先写老方案，再在 @supports 里升级 ​

css
.panel {
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.3s ease;
}

.panel.is-open {
  max-height: 40rem;
}

@supports (interpolate-size: allow-keywords) {
  :root {
    interpolate-size: allow-keywords;
  }

  .panel {
    max-height: none;
    height: 0;
    overflow: clip;
    transition: height 0.3s ease;
  }

  .panel.is-open {
    height: auto;
  }
}

这是线上项目最稳的接法之一。

技巧三：FAQ、折叠区优先用真实尺寸动画，不要默认上 scaleY ​

css
.answer {
  height: 0;
  overflow: clip;
  transition: height 0.25s ease;
}

如果内容本来就在文档流里，真实尺寸变化通常比缩放伪装更自然、更可维护。

技巧四：配合 opacity 做更柔和的展开体验 ​

css
.panel {
  height: 0;
  opacity: 0;
  overflow: clip;
  transition: height 0.28s ease, opacity 0.18s ease;
}

.panel.is-open {
  height: auto;
  opacity: 1;
}

这样不仅尺寸平滑，内容出现也不会那么生硬。

技巧五：给展开内容预留一点内边距动画感 ​

css
.panel {
  height: 0;
  overflow: clip;
  padding-block: 0;
  transition: height 0.26s ease, padding-block 0.26s ease;
}

.panel.is-open {
  height: auto;
  padding-block: 12px;
}

非常适合 FAQ、设置面板、注释块。视觉上会更“呼吸感”。

技巧六：动态内容较多时，把展开区域包一层内壳 ​

css
.panel {
  height: 0;
  overflow: clip;
}

.panel-inner {
  padding: 12px 16px;
}

这样在复杂内容布局下更容易控制动画裁切与内边距关系，减少穿帮。

十二、常见误区 ​

1. 以为它能解决所有尺寸动画问题 ​

不能。它主要解决的是内在尺寸关键字参与插值的问题，尤其适合 auto 这类场景。复杂布局联动、跨多个元素的同步动画，仍然可能需要 JS。

2. 忘记设置裁切 ​

如果展开区域没有 overflow: clip 或 hidden，内容在高度过渡过程中很容易提前露出来，视觉会很乱。

3. 只关注展开，不测试收起 ​

很多人只看展开效果，忽略收起是否顺滑。真正好用的折叠组件，必须：

  • 展开自然
  • 收起也自然
  • 内容动态变化后仍然稳定

4. 在固定高度场景硬上 ​

如果一个区域本来就应该有固定高度，比如统一卡片高度、固定工具条，那就没必要为了“新特性”去引入尺寸插值。

5. 误以为不需要兼容策略 ​

现代能力再好，真实业务里还是得考虑旧浏览器或弱兼容环境。@supports 是非常值得保留的一层保险。

十三、总结 ​

interpolate-size 的出现，解决了一个困扰前端很久的问题：让 auto 这类内在尺寸值也能更合理地参与动画过渡。

它尤其适合：

  • FAQ
  • 手风琴
  • 折叠菜单
  • 评论展开区
  • 设置面板
  • 任何“内容高度不固定但又想平滑开合”的组件

你可以把它这样理解：

  • 传统 CSS 动画更擅长数值到数值
  • auto 过去常常是动画死角
  • interpolate-size 让这个死角开始被原生 CSS 接管

在工程实践里，最推荐的策略是：

  • 基础方案先可用
  • 现代浏览器用 @supports 增强
  • 需要时再保留 max-height 或 JS 回退

如果你经常做后台、文档站、内容产品、交互组件库，这个能力非常值得尽早掌握。它不是那种“看起来很炫”的新特性，但会实实在在减少你为尺寸动画写的很多笨重补丁代码。
